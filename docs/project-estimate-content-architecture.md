# Смета проекта — хранение разделов, работ и материалов

> Last updated: 2026-05-20
>
> Status: рабочий слой редактора сметы.
>
> Related issues: #142, #149, #151.

## Назначение

Этот слой хранит состав выбранной сметы проекта: разделы, работы и материалы.

Главная запись сметы остаётся в таблице:

```txt
project_estimate_records
```

Новая структура не создаёт отдельную таблицу смет и не заменяет реестр смет проекта.

## Структура

```txt
project_estimate_records
  -> project_estimate_sections
      -> project_estimate_works
          -> project_estimate_materials
```

## Разделы

Таблица:

```txt
project_estimate_sections
```

Раздел всегда принадлежит:

```txt
рабочей области
проекту
одной смете из project_estimate_records
```

Основные поля:

```txt
title
number
sortOrder
worksAmount
materialsAmount
totalAmount
```

Обычный рабочий экран должен показывать только строки без `archivedAt` и `deletedAt`.

Создание раздела в редакторе использует автоматический номер. Пользователь вводит только название.

## Работы

Таблица:

```txt
project_estimate_works
```

Работа всегда принадлежит разделу.

Работа может быть добавлена из справочника или вручную. Основной пользовательский сценарий в редакторе — добавление из справочника работ. Если работа добавлена из справочника, в строку сметы копируются рабочие значения на момент добавления:

```txt
code
title
unitCode
unitLabel
price
category
```

Связь со справочником необязательная:

```txt
directoryWorkId
directoryWorkVersion
```

Изменение справочника работ позже не должно менять уже сохранённую смету.

Сумма работы хранится в строке:

```txt
totalAmount = quantity * price
```

В редакторе работа может быть заменена через справочник работ. При замене меняются название и цена работы. Количество работы и материалы внутри работы не меняются.

## Коэффициент работ

В смете есть коэффициент работ. Он хранится в главной записи сметы:

```txt
project_estimate_records.works_coefficient_percent
```

Для работы хранится базовая цена:

```txt
project_estimate_works.base_price
```

Правило расчёта:

```txt
price = base_price * (1 + works_coefficient_percent / 100)
```

После расчёта цена работы округляется вверх до ближайших 10 ₽:

```txt
92 + 10% = 101.2 -> 110
100 + 10% = 110 -> 110
101 + 10% = 111.1 -> 120
```

Коэффициент применяется только к работам. Материалы не меняются.

Окно коэффициента открывается существующей кнопкой `Коэффициент` в верхнем `ButtonGroup` тулбара. При открытии окна пользователь видит текущий сохранённый коэффициент.

Сейчас пересчёт коэффициента работает построчно. Ускорение пересчёта для больших смет вынесено в issue #151.

## Материалы

Таблица:

```txt
project_estimate_materials
```

Материал всегда принадлежит работе. Материал не добавляется напрямую в смету без работы.

Материал может быть добавлен из справочника или вручную. Основной пользовательский сценарий в редакторе — добавление из справочника материалов внутрь конкретной работы. Если материал добавлен из справочника, в строку сметы копируются рабочие значения на момент добавления:

```txt
code
title
unitCode
unitLabel
price
supplierName
```

Связь со справочником необязательная:

```txt
directoryMaterialId
directoryMaterialVersion
```

Изменение справочника материалов позже не должно менять уже сохранённую смету.

## Расход материала

В этом слое `consumption` означает расход, а не процент отхода.

Хранятся оба значения:

```txt
quantity
consumption
```

Пользовательское правило:

```txt
количество материала = количество работы / расход
```

Обратное правило:

```txt
расход = количество работы / количество материала
```

Сумма материала:

```txt
totalAmount = quantity * price
```

Расход влияет на количество материала, а не напрямую на сумму.

Поле `waste` в первом слое не используется.

## Итоги

Для быстрых итогов хранятся суммы:

```txt
раздел: worksAmount, materialsAmount, totalAmount
работа: totalAmount
материал: totalAmount
смета: amount в project_estimate_records
```

База пересчитывает:

```txt
работа = quantity * price
материал = quantity * price
раздел = сумма активных работ и материалов
project_estimate_records.amount = общий итог всех активных разделов
```

Архивные и удалённые строки не участвуют в обычных итогах.

## Порядок строк

На каждом уровне есть явный порядок:

```txt
project_estimate_sections.sort_order
project_estimate_works.sort_order
project_estimate_materials.sort_order
```

Пользовательское изменение порядка строк в UI пока не реализовано. Эта доработка вынесена в issue #151.

## Действия редактора

В текущем рабочем слое поддержаны:

```txt
создание раздела
добавление работы из справочника
замена работы через справочник
добавление материала из справочника
редактирование названия, количества и цены работы
редактирование названия, количества, расхода и цены материала
удаление раздела
удаление работы
удаление материала
коэффициент работ
```

Удаление раздела, работы и материала выполняется только через `Dialog` подтверждения. `window.confirm` не используется.

Серверный кэш для содержимого редактора сметы отключён. Рабочий документ должен читаться свежим после изменений. Кэш может использоваться только для вспомогательных списков выбора работ и материалов.

## Безопасность данных

Все таблицы содержат `workspace_owner_id`.

Связи построены так, чтобы строка из другой рабочей области или другого проекта не могла быть привязана к смете.

Чтение доступно участникам текущей рабочей области. Изменение доступно разрешённым ролям текущей рабочей области.

Клиент не должен передавать рабочую область как источник истины. Серверный слой определяет рабочую область по текущему пользователю.
