# Управление доступом (Access Control / RBAC)

> 2026-05-23 · статус: полностью реализован (production бэкенд и интерактивный фронтенд с визуализацией матрицы прав доступа).

---

## Назначение

Ролевая модель управления доступом (RBAC) для multi-tenant workspace.
5 ролей × 19 прав = полная матрица доступа. Реализована на трёх уровнях:
1. База данных (таблицы, связи, индексы, RLS).
2. Серверная логика (`lib/auth/permissions.ts`, Server Actions).
3. Клиентский интерфейс (интерактивная таблица матрицы прав доступа).

## Данные (таблицы, типы)

### Таблицы БД (схема public)

#### `roles` — определения ролей
Описывает роли: `owner` (Владелец), `admin` (Администратор), `manager` (Менеджер), `estimator` (Сметчик), `viewer` (Наблюдатель). Системные роли отмечены флагом `locked` и защищены от удаления.

#### `permissions` — определения прав (19 записей, 5 групп)
Содержит права по направлениям: `billing`, `estimates`, `projects`, `purchases`, `team`.

#### `role_permissions` — связи роль→право (61 запись)
Связывает роли и права. Задает дефолтную матрицу доступа.

#### `user_roles` — связи пользователь↔роль
Назначение ролей пользователям в конкретном workspace.

## API и Server Actions

- `GET /api/team/roles` — Список ролей с привязанными правами.
- `POST /api/team/roles` — Создание кастомных ролей с набором прав (`team.manage`).
- `GET /api/team/members` — Список участников текущего workspace и их назначенных ролей.
- `POST /api/team/members` — Назначение роли пользователю.
- `DELETE /api/team/members` — Снятие роли с пользователя (запрещено для locked-ролей).

Хуки `useAssignRole` и `useRemoveRole` используют Server Actions из `app/actions/access-control` для мутаций данных.

---

## Компоненты UI

### `PermissionsMatrix` (Матрица прав)
Отображает таблицу ролей и прав. Позволяет наглядно просматривать, какие возможности доступны каждой из ролей. Роль Владельца (`owner`) заблокирована для редактирования, так как всегда обладает полными правами.
